# AGENTS.md

Project: **notagrafo** (`NOTA`)

This Mnema project is managed by the `@felipesauer/mnema` MCP server. Anything an agent does in this repository must flow through the tools exposed by the server.

## Operating principles

1. Start every session with the `context_bootstrap` tool.
2. Wrap every batch of mutations in `agent_run_start` / `agent_run_end`.
3. Prefer transition tools (e.g. `task_submit`) over editing fields directly.
4. Read first, write second. The audit log records every mutation, including yours.
5. Satisfy a transition gate тАФ supply every required field. Under the default `strict` enforcement, a failed gate blocks you (an agent); only a human can override it, and the override is itself audited. Do not try to route around a gate; complete the fields it asks for.

## Workflow: `default`

Default 7-state workflow: DRAFT тЖТ READY тЖТ IN_PROGRESS тЖТ IN_REVIEW
тЖТ DONE, with BLOCKED and CANCELED branches. Submitting a draft
requires title, description, acceptance criteria and an estimate.
Code review is mandatory before DONE.

## Dual identity

Every mutation captures three identifiers: `actor` (the human responsible), `via` (the agent that did the work, sourced from MCP `agent_handle`), and `run` (the run id from `agent_run_start`). Mutations without an active run are rejected with `NO_ACTIVE_RUN`.

## MCP tool surface

The tools are organised in conceptual layers тАФ reason about these buckets rather than memorising every tool. `context_bootstrap` returns the exact per-tool grouping (`tool_groups`) for this project:

- **Core** тАФ audit, agent runs/plans, tasks, dependencies, evidence, search and read-only graph/snapshot. Always available.
- **Workflow transitions** тАФ one `task_<action>` per workflow transition.
- **Planning** тАФ epics, sprints and their coverage/lint. Available when the active workflow enables epics and/or sprints.
- **Knowledge** тАФ decisions/ADRs, skills, memories, observations and the provenance/wikilink chain. Available in this project.

When a layer is available, prefer its MCP tools over the CLI mid-run, and prefer the batch forms тАФ `task_create_many`, `sprint_add_tasks`, `task_depends_many` тАФ which attempt every item and report per-item failures instead of failing the whole call.

## Recording what you learn

**Use Mnema for this, not your own memory.** If your client has a built-in memory feature (a personal notes file, native recall), do **not** put durable facts about *this project* there: those stay on your machine, never reach a teammate, and leave no audit trail. Record them through the Mnema tools below тАФ they are mirrored to `.md` in the repo and recorded in the hash-chained log, so the knowledge travels with the project and is provable. Your native memory is still fine for your own cross-project habits; project knowledge belongs in Mnema.

These are not optional housekeeping тАФ they are how the next session (yours or a teammateтАЩs) avoids relearning what you already know. Record as you work, not in a batch at the end. Concretely:

- **Hit a non-obvious fact about *this* project** (a constraint, a convention, why something is the way it is)? Write a **memory** with `memory_record(slug, title, content, тАж)`. Upsert by slug тАФ the latest content wins. Rule of thumb: if you would re-explain it to yourself next week, it is a memory.

- **Worked out a repeatable procedure** (a sequence of steps you would follow again)? Write a **skill** with `skill_record(slug, name, description, content, тАж)`, and call `skill_use` each time you actually apply it so the useful ones rise. Use `mode='new_version'` for a disruptive rewrite; the default updates in place. An empty `skills/` after real work usually means this step was skipped тАФ donтАЩt.

- **Noticed something that might matter later** but isnтАЩt yet a durable fact (a smell, a surprise, a TODO-shaped signal)? Append an **observation** with `observation_record` тАФ no slug, no upsert, fire-and-forget. It is the cheapest gateway: an observation can later be consolidated into a memory or hardened into a skill.

- **Made a choice the team should be able to contest later**? Record a formal ADR with `decision_record`. When a free-form `note_add` matures into a decision, use `decision_promote_from_note(note_id, тАж)` so the audit log links the ADR back to the note.

All of the above are mirrored to `.md` on disk so they travel with the repository and are reviewable in a pull request. The `memory_index` / `decisions_index` fields in `context_bootstrap` are human-curated supplements regenerated by `mnema memory consolidate` тАФ read them for context, never write to them directly.
