# AGENTS.md

Project: **notagrafo** (`NOTA`)

This Mnema project is managed by the `@felipesauer/mnema` MCP server. Anything an agent does in this repository must flow through the tools exposed by the server.

## Operating principles

1. Start every session with the `context_bootstrap` tool.
2. Wrap every batch of mutations in `agent_run_start` / `agent_run_end`.
3. Prefer transition tools (e.g. `task_submit`) over editing fields directly.
4. Read first, write second. The audit log records every mutation, including yours.
5. Satisfy a transition gate â€” supply every required field. Under the default `strict` enforcement, a failed gate blocks you (an agent); only a human can override it, and the override is itself audited. Do not try to route around a gate; complete the fields it asks for.

## Dual identity

Every mutation captures three identifiers: `actor` (the human responsible), `via` (the agent that did the work, sourced from MCP `agent_handle`), and `run` (the run id from `agent_run_start`). Mutations without an active run are rejected with `NO_ACTIVE_RUN`.

## MCP tool surface

Tools are organised in conceptual layers â€” reason about these buckets, not every tool; `context_bootstrap` returns the exact per-tool grouping (`tool_groups`):

- **Core** â€” audit, agent runs/plans, tasks, dependencies, evidence, search and read-only graph/snapshot. Always available.
- **Workflow transitions** â€” one `task_<action>` per workflow transition.
- **Planning** â€” epics, sprints and their coverage/lint. Available when the active workflow enables epics and/or sprints.
- **Knowledge** â€” decisions/ADRs, skills, memories, observations and the provenance/wikilink chain. Available in this project.

Prefer MCP tools over the CLI mid-run, and the batch forms (`task_create_many`, `sprint_add_tasks`, `task_depends_many`) â€” they report per-item failures instead of failing the whole call.
