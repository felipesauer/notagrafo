MERGE DOS 7 PRs EMPILHADOS (Sprints 1-7) — armadilha e recuperação. A cadeia era linear (#2→main, #3..#8 cada um sobre o anterior). Mergeei via gh re-apontando a base de cada PR para main em sequência rápida (gh pr edit --base main; gh pr merge --merge). RESULTADO RUIM: o GitHub marcou os 7 como MERGED, mas só o conteúdo do #2 chegou na main — os #3..#8 mergearam diffs essencialmente vazios porque cada um foi re-baseado e mergeado antes do anterior estar de fato na main (corrida). Sintoma: origin/main tinha 11 commits e só a Sprint 1; faltavam CONTRIBUTING/docker-compose/ci.yml/Grafo/server.ts/seed. RECUPERAÇÃO (conteúdo nunca se perdeu — estava intacto em origin/feat/sprint-7-infra, 073923b): git checkout main + reset --hard origin/main, git merge origin/feat/sprint-7-infra (traz Sprints 2-7 de uma vez, exit 0), sanity (install/build/typecheck/lint/test:unit 36/36), git push origin main. Origin/main final: 62 commits, 5 pacotes, todos os arquivos-chave do MVP, 0 PRs abertos. LIÇÃO: para PRs empilhados lineares, NÃO re-apontar+mergear em sequência rápida — ou esperar cada merge propagar de verdade na base antes do próximo, ou simplesmente fazer um único merge/FF da branch da ponta na main (a ponta já contém toda a cadeia linear). Cuidado também: pnpm install mexe no estado e um arquivo untracked do mnema (.mnema/backlog/DONE/*.md) bloqueia git checkout (usar git stash -u antes).
