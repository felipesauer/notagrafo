DÍVIDA — suíte e2e (Playwright contra docker compose) instável no CI, observada na auditoria 2026-06-30 (PRs #21/#22). DOIS sintomas distintos, ambos ALHEIOS ao código de produção dos PRs:

(1) FRÁGIL/determinístico — companies.spec.ts:8 (e provavelmente products/overview/nf-detail.spec): `getByRole('heading', {name: /empresas|companies/i})` casa com 2 elementos — o <h1 class="header__breadcrumb"> (Header.tsx:22, mostra o título da rota) E o <h2> da página → 'strict mode violation'. Fix: tornar o seletor específico (ex.: page.locator('h2', {hasText: ...}) ou getByRole('heading',{level:2})). Afeta 4 specs que usam o mesmo padrão.

(2) FLAKY/boot — `docker compose --profile app up -d --wait` falha com 'container notagrafo-api-1 is unhealthy': a API esgota o healthcheck enquanto espera o Neo4j ficar healthy (~30s de janela; Neo4j é o mais lento a subir). Fix: aumentar start_period/retries do healthcheck da api no docker-compose, ou depends_on: condition: service_healthy do neo4j com timeout maior.

Os PRs #21 (A0/A1) e #22 (A2/A3/A4) têm lint/unit/integração verdes; só o e2e oscila por esses dois motivos. Não bloqueia o mérito das correções. Vale uma task dedicada para estabilizar o e2e.
